01MODELAR

// CLASSE NÃO É APENAS UMA CAIXA

Use objetos quando regras pertencem aos dados.

Um dicionário representa um registro com flexibilidade. Uma classe passa a valer a pena quando o conceito tem identidade, operações válidas e invariantes que precisam continuar verdadeiras.

STRUCTURE

Dicionário

equipment = {
    "id": "PUMP-014",
    "tests": [],
}

Bom para troca de dados e estruturas simples. Qualquer parte do programa pode alterar qualquer chave.

→
BEHAVIOR

Objeto

equipment = Equipment(
    "PUMP-014", "pump"
)
equipment.add_test(2, 0.91)

A criação e as mudanças passam por métodos que validam o domínio.

REGRA PRÁTICA

Não crie classes apenas para agrupar campos. Crie uma classe quando ela puder responder por comportamento e consistência.

02ANATOMIA

// DO MOLDE À INSTÂNCIA

Classe define o contrato.
Objeto vive o estado.

Cada instância recebe seus próprios atributos. Métodos são funções definidas na classe; `self` referencia a instância que recebeu a chamada.

class Equipment:
    """Aggregate equipment test results."""

    def __init__(self, equipment_id, category):
        self.equipment_id = equipment_id
        self.category = category
        self._tests = []
CLASS

Namespace e contrato compartilhado.

__INIT__

Inicializa uma instância já criada.

SELF

Objeto que recebe a operação.

INSTANCE METHOD

Opera sobre `self`

Lê ou modifica o estado de uma instância.

equipment.add_test(...)
CLASS METHOD

Opera sobre `cls`

Útil para construtores alternativos e regras da classe.

Equipment.from_dict(data)
STATIC METHOD

Sem `self` ou `cls`

Função relacionada ao conceito, mas independente do estado.

Equipment.valid_status(2)
03REGRAS

// PROTEJA AS INVARIANTES

Mudar o estado deve ser uma operação válida.

Python confia em convenções. O prefixo `_` comunica uso interno; propriedades expõem leitura calculada ou validada sem transformar cada acesso em chamada explícita.

INPUTstatus=5

Dado recebido de fora.

→
VALIDATION1 ≤ status ≤ 4

A regra vive perto do objeto criado.

→
RESULTValueError

Estado inválido nunca é armazenado.

PRIVATE BY CONVENTION

`self._tests`

A lista interna pode mudar, mas não deve ser entregue diretamente para alteração externa.

READ-ONLY VIEW

`tests` property

Retornar uma tupla oferece leitura sem compartilhar a coleção mutável original.

PROPERTY NÃO É ESCONDER CUSTO

Uma propriedade deve parecer um acesso simples. Operações lentas, I/O e mudanças importantes continuam melhores como métodos explícitos.

04RELAÇÕES

// COMPOR ANTES DE HERDAR

“Tem um” e “é um” não são a mesma relação.

Composição conecta objetos por colaboração. Herança especializa um contrato. A primeira costuma manter responsabilidades mais claras e mudanças mais locais.

COMPOSITION · HAS-A
EquipmentcontainsTestResult[]

Equipamento tem resultados

`Equipment` coordena objetos `TestResult` imutáveis. Cada classe preserva uma responsabilidade.

INHERITANCE · IS-A
EquipmentextendsCalibratedEquipment

Use quando existe substituição real

A subclasse deve cumprir o contrato da base. Reuso de código sozinho não justifica herança.

@DATACLASS

Menos repetição para objetos de valor.

`TestResult` precisa principalmente de dados, validação, igualdade e representação. `frozen=True` torna a intenção de imutabilidade explícita.

@dataclass(frozen=True, slots=True)
class TestResult:
    health_status: int
    confidence: float

    def __post_init__(self):
        if self.health_status not in range(1, 5):
            raise ValueError("invalid status")
HERANÇA É UM CONTRATO

Se uma subclasse precisa desativar ou contradizer comportamentos da classe base, provavelmente a relação foi modelada de forma incorreta.
