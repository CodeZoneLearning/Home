PY_SYSTEMS · API CONTRACTS

API é contrato.
Python executa.

Uma API define como sistemas conversam: qual recurso existe, quais dados entram, quais respostas saem e como erros são comunicados. Antes de escolher FastAPI, Flask ou qualquer biblioteca, entenda o contrato.

01CONTRATO

// FRONTEIRA ENTRE SISTEMAS

API não é só endpoint. É uma promessa operacional.

Quando alguém usa sua API, essa pessoa passa a depender de nomes, formatos, status e comportamento. Se o contrato muda sem cuidado, o sistema consumidor quebra mesmo que o seu código esteja “funcionando”.

ENTRADA

O que pode ser enviado

Parâmetros, corpo da requisição, headers, autenticação, tipos e campos obrigatórios.

PROCESSO

O que a API faz

Regra executada, validações aplicadas, limites, efeitos colaterais e idempotência.

SAÍDA

O que será devolvido

Status HTTP, payload, mensagens de erro, paginação, links e formato de datas.

MENTALIDADE

Python pode consumir uma API, criar uma API ou orquestrar chamadas entre serviços. Em todos os casos, o ponto central é respeitar o contrato de dados.

02HTTP

// REQUISIÇÃO E RESPOSTA

HTTP dá vocabulário para a conversa.

A maioria das APIs web usa HTTP. Isso significa que métodos, URLs, headers, body e status code carregam significado. Usar esses elementos com clareza reduz documentação ambígua.

METHOD

Verbo

`GET` lê, `POST` cria ou executa, `PUT/PATCH` atualizam e `DELETE` remove. O verbo comunica intenção.

RESOURCE

Recurso

URLs devem representar coisas do domínio, como `/equipments` e `/tests`, não nomes internos de funções.

STATUS

Resultado

`200`, `201`, `400`, `401`, `404` e `500` dizem rapidamente se a requisição funcionou e por quê.

STATUS_LAB

Selecione um cenário.

RESPOSTA ESPERADA200 OK

A requisição foi aceita e a API devolveu o recurso solicitado.

03DESIGN

// DADOS PRECISAM DE FORMA

Schema reduz interpretação livre.

Um bom contrato declara campos, tipos, exemplos, obrigatoriedade e erros. Isso permite validar cedo, gerar documentação e criar testes de contrato.

Documentação que não acompanha o comportamento real atrapalha. O ideal é gerar ou validar documentação a partir de schemas e testes.

04PYTHON

// CLIENTE E SERVIDOR

Python aparece dos dois lados do contrato.

Como cliente, Python envia requisições e trata respostas. Como servidor, Python recebe dados, valida, executa regras e devolve um contrato previsível.

CONSUMIR API

Cliente

response = requests.get(url, timeout=10)
response.raise_for_status()
data = response.json()

O cliente precisa lidar com timeout, status inesperado e campos ausentes.

CRIAR API

Servidor

@app.get("/equipments/{equipment_id}")
def read_equipment(equipment_id: str):
    return {"equipment_id": equipment_id}

O servidor precisa validar entrada, controlar erro e manter resposta estável.

REGRA DE ENGENHARIA

Antes de escrever endpoint, escreva exemplos de request e response. Se o contrato não está claro em papel, o código vai esconder ambiguidade.

05CUIDADOS

// O QUE QUEBRA EM PRODUÇÃO

APIs falham nas fronteiras.

A parte difícil não é criar uma rota que responde. É sustentar comportamento previsível quando dados chegam errados, rede falha, usuário não tem permissão ou consumidores dependem de versões antigas.

Valide entrada antes da regra

Retorne erro claro para campo ausente, tipo inválido ou valor fora do domínio.

400

Proteja identidade e autorização

Autenticação diz quem é a pessoa ou sistema. Autorização diz o que pode fazer.

401/403

Defina timeouts e limites

Sem limites, uma API lenta pode travar consumidores e multiplicar falhas.

SLO

Não quebre contrato sem versão

Campos removidos, formatos alterados e novos significados precisam de migração.
