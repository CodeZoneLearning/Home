02VENV + PIP

// FLUXO DA BIBLIOTECA PADRÃO

Crie o ambiente com o próprio Python.

O módulo `venv` cria um ambiente a partir do Python escolhido. Depois, use `python -m pip` para garantir que o instalador pertence ao mesmo interpretador.

01 · CREATEpython -m venv .venv

Cria o diretório local do ambiente.

02 · ACTIVATE.venv\Scripts\Activate.ps1

No PowerShell, ajusta o `PATH` da sessão atual.

03 · INSTALLpython -m pip install -r requirements.txt

Instala as dependências declaradas.

04 · RUNpython -m equipment_health.main

Executa usando o ambiente ativo.

INTERPRETADOR PRIMEIRO

Verifique com `python -c "import sys; print(sys.executable)"`. O caminho deve apontar para `.venv` antes de instalar ou executar.

03UV

// PROJETO, AMBIENTE E LOCK

`uv` coordena o fluxo do projeto.

Em um projeto gerenciado, `uv` lê o `pyproject.toml`, resolve dependências, mantém um lockfile e executa comandos dentro do ambiente do projeto.

INITuv init equipment-health

Cria a estrutura e a declaração inicial do projeto.

→
ADDuv add rich

Declara a dependência e atualiza a resolução.

→
RUNuv run equipment-health

Sincroniza quando necessário e executa no ambiente.

→
SYNCuv sync

Reconstrói o ambiente a partir do projeto e do lock.

Você ainda tem uma `.venv`

O `uv` normalmente usa um ambiente de projeto. A diferença é que criação, sincronização e execução ficam coordenadas.

Ativação deixa de ser obrigatória

`uv run` seleciona o ambiente do projeto sem depender do `PATH` previamente ativado.

LOCK NÃO SUBSTITUI `PYPROJECT`

`pyproject.toml` descreve as dependências permitidas; `uv.lock` registra uma resolução exata para reprodução consistente. Em aplicações, normalmente versione ambos.

A estrutura do projeto separa código-fonte, declaração de dependências, resolução e arquivos locais descartáveis.

Metadados e dependências diretas.

COMMITuv.lock

Resolução reproduzível do projeto.

IGNORE.venv/

Artefato local reconstruível.

DECIDE.python-version

Útil para alinhar a versão do interpretador.

UM FLUXO PRINCIPAL

Evite atualizar manualmente `requirements.txt` e `pyproject.toml` como duas fontes independentes. Defina qual arquivo governa o projeto e gere os demais quando necessário.
