01MENTALIDADE

// NÃO PEÇA MÁGICA; DEFINA TRABALHO

O modelo responde melhor quando a tarefa é observável.

Prompts úteis reduzem ambiguidade. Eles explicam o papel do modelo, o objetivo da tarefa, o contexto permitido, o formato de resposta e os critérios que tornam a resposta boa ou ruim.

01 · TAREFA

O que precisa ser feito?

Classificar, resumir, extrair, comparar, transformar, revisar ou gerar uma resposta. Verbo claro reduz saída genérica.

02 · EVIDÊNCIA

Com base em quais dados?

Inclua o contexto relevante e diga se o modelo pode inferir, perguntar ou recusar quando faltar informação.

03 · SAÍDA

Como a resposta será usada?

Texto para leitura humana pode ser flexível. Saída para código, filtro ou banco precisa de schema e validação.

REGRA CENTRAL

Prompt é parte do sistema. Se a resposta alimenta outro processo, o prompt deve declarar formato, limites e tratamento de erro.

02DICAS

// DICAS DE OURO

Seis hábitos que melhoram quase qualquer prompt.

Essas regras não são truques. Elas tornam o trabalho verificável, reduzem surpresa e facilitam comparar versões do prompt.

GOLD_01

Comece pelo resultado

Diga qual decisão a resposta deve apoiar. Um resumo para estudo é diferente de um resumo para aprovação executiva.

GOLD_02

Dê contexto, não ruído

Inclua dados necessários, definições e restrições. Remova texto que não muda a resposta.

GOLD_03

Mostre o formato esperado

Use bullets, tabela, JSON ou campos nomeados. O formato é parte da instrução.

GOLD_04

Declare critérios de qualidade

Explique o que conta como boa resposta: precisão, concisão, evidência, tom, risco ou completude.

GOLD_05

Inclua exemplos curtos

Um exemplo certo e um errado alinham estilo, granularidade e limites melhor que uma explicação longa.

GOLD_06

Defina o que fazer quando faltar dado

“Se não houver evidência, responda não encontrado” é melhor do que deixar o modelo completar lacunas.

03ANATOMIA

// DO FRACO AO OPERACIONAL

Compare um pedido solto com uma especificação.

O mesmo objetivo pode produzir respostas muito diferentes. A diferença está na clareza de papel, contexto, formato e critérios.

PROMPT FRACO

“Resume esse texto.”

O modelo não sabe tamanho, público, finalidade, tópicos obrigatórios, nível técnico ou o que ignorar.

PROMPT MELHOR

“Resuma para decisão técnica.”

Você é um analista. Use apenas o texto abaixo. Gere 5 bullets com: problema, causa, evidência, risco e próxima ação. Se faltar dado, escreva “não informado”.

ROLE

Define a perspectiva: analista, tutor, revisor, classificador, extrator.

TASK

Define a operação: classificar, resumir, extrair, comparar ou transformar.

CONTEXT

Define a evidência que pode ser usada e o que deve ser ignorado.

CONSTRAINTS

Define limites de tom, tamanho, segurança, recusa e incerteza.

OUTPUT

Define campos, tipos, ordem, exemplos e formato final.

TESTE PRÁTICO

Se duas pessoas lerem seu prompt e imaginarem respostas completamente diferentes, ele ainda não é uma especificação.

04SAÍDA

// TEXTO LIVRE É DIFÍCIL DE INTEGRAR

Saída estruturada transforma resposta em dado.

Quando a resposta será consumida por outro código, dashboard, filtro ou banco de dados, prefira um contrato explícito. Isso permite validar, rejeitar e monitorar respostas quebradas.

JSON SIMPLES

Bom para protótipo

{
  "categoria": "billing",
  "prioridade": 2,
  "confianca": 0.82
}
SCHEMA

Bom para produto

{
  "type": "object",
  "required": ["categoria", "prioridade"],
  "properties": {
    "prioridade": { "type": "integer" }
  }
}
01

Campos previsíveis

Use nomes estáveis. Evite sinônimos em cada resposta.

02

Tipos claros

Número, texto, booleano, lista e objeto precisam ter significado definido.

03

Enumerações

Quando possível, limite valores aceitos: `alta`, `media`, `baixa`.

04

Fallback explícito

Defina `null`, `unknown` ou erro quando faltar evidência.

NÃO CONFIE SEM VALIDAR

Mesmo quando o prompt pede JSON, valide a resposta antes de usar. Prompt reduz erro, validação protege o sistema.

06CHECKLIST
// ANTES DE APROVAR
Checklist para prompts de produção.
Use estes pontos antes de transformar uma ideia em componente de produto.

      
01
A tarefa tem verbo claro?
Evite “analise”. Prefira “classifique”, “extraia”, “compare” ou “gere”.
02
O contexto permitido está explícito?
O modelo sabe se pode inferir, perguntar, recusar ou usar conhecimento geral?
03
Existe formato de saída validável?
Se outro sistema consome a resposta, use JSON, schema e validação.
04
Há exemplos e contraexemplos?
Exemplos reduzem ambiguidade de estilo, granularidade e limites.
05
O erro esperado está definido?
Determine o que fazer quando faltam dados, há conflito ou a confiança é baixa.
