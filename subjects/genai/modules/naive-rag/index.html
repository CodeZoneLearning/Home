RAG_BASELINE · GROUNDING

RAG primeiro.
Framework depois.

Naive RAG é o baseline mais simples para conectar um LLM a conhecimento externo: buscar trechos relevantes, colocar no prompt e pedir uma resposta com base nas fontes. Ele não resolve tudo, mas cria uma arquitetura clara para aprender.

01PROBLEMA

// MEMÓRIA DO MODELO NÃO É BASE DE DADOS

RAG existe para reduzir distância entre resposta e evidência.

Um LLM conhece padrões aprendidos no treinamento, mas não tem acesso garantido aos seus documentos internos, versões atuais, políticas, PDFs ou bases de conhecimento. RAG adiciona uma etapa de recuperação antes da geração.

01 · PARAMÉTRICO

O que o modelo já aprendeu

Conhecimento geral, linguagem, padrões e raciocínios aproximados. É útil, mas não prova que a resposta veio de uma fonte específica.

02 · EXTERNO

O que você recupera

Trechos de documentos, tabelas, manuais, políticas ou artigos que podem ser citados e atualizados sem treinar o modelo novamente.

03 · GROUNDED

O que a resposta deve usar

A saída precisa estar limitada ao contexto recuperado. Quando a evidência não aparece, a resposta correta pode ser recusar ou pedir mais dados.

OBJETIVO

RAG não é “fazer o modelo saber mais”. É criar um processo em que a resposta pode apontar para evidências recuperadas e ser avaliada.

02PIPELINE

// O BASELINE TEM SEIS PEÇAS

Naive RAG é uma esteira simples de recuperação e geração.

A versão ingênua não usa grafo, agente, memória longa ou roteamento sofisticado. Ela serve para validar se a base documental e a busca já conseguem sustentar respostas úteis.

01

Ingerir

Carregue documentos e preserve metadados como origem, data, seção e versão.

02

Chunkar

Divida texto em unidades pequenas o bastante para buscar, mas completas o bastante para fazer sentido.

03

Embeddar

Transforme chunks e perguntas em vetores comparáveis por similaridade.

04

Buscar

Recupere os trechos mais próximos da pergunta, normalmente usando top-k.

05

Montar prompt

Inclua pergunta, contexto, regra de uso das fontes e formato esperado.

06

Responder

Gere a resposta e cite quais trechos sustentam a conclusão.

BASELINE ANTES DA COMPLEXIDADE

Se o Naive RAG falha porque nenhum trecho relevante aparece no top-k, trocar o prompt dificilmente resolve. O problema está antes da geração.

03FALHAS

// RAG TAMBÉM ERRA

Quando a resposta parece boa, mas o processo está fraco.

O erro pode nascer no documento, no chunk, na busca, no prompt ou na própria síntese. Separar essas etapas evita culpar o LLM por um problema de recuperação.

RETRIEVAL

Busca traz o trecho errado

A resposta fica bem escrita, mas baseada em contexto irrelevante. Métrica útil: recall dos trechos esperados.

CHUNKING

Contexto foi quebrado demais

O trecho recuperado contém uma frase útil, mas perdeu a condição, exceção ou tabela que dava significado.

GENERATION

Modelo extrapola a fonte

Mesmo com bons trechos, o modelo pode completar lacunas. Regra útil: responder “não encontrado no contexto”.

CUIDADO

Uma resposta com citação não é automaticamente correta. A fonte citada precisa realmente sustentar a frase que foi escrita.

04DECISÕES

// ESCOLHAS QUE MUDAM O RESULTADO

Todo componente do RAG carrega uma hipótese.

O baseline deve registrar as decisões para que você consiga comparar versões. Sem isso, cada mudança parece melhoria, mas ninguém sabe o que mudou de fato.

CHUNK

Tamanho e sobreposição

Chunks menores melhoram precisão local. Chunks maiores preservam contexto. Overlap reduz cortes ruins, mas aumenta ruído e custo.

SEARCH

Top-k e threshold

Poucos trechos podem perder evidência. Muitos trechos confundem o prompt. Threshold ajuda a recusar quando a busca é fraca.

PROMPT

Contrato da resposta

Declare que o modelo deve usar apenas o contexto, citar fontes e admitir ausência de evidência quando necessário.

REGRA PRÁTICA

Antes de adicionar framework, salve perguntas de teste, chunks recuperados e respostas esperadas. Esse conjunto vira o seu primeiro dataset de avaliação.

06CHECKLIST

// ANTES DE CHAMAR DE PRODUTO

Um RAG útil precisa ser auditável.

A primeira versão pode ser simples, mas deve deixar rastros suficientes para você explicar de onde veio a resposta e por que ela deveria ser confiável.

01

Mostre as fontes usadas

Resposta sem referência vira opinião do modelo. Resposta com referência pode ser checada.

02

Teste perguntas que deveriam falhar

Um bom sistema sabe recusar quando a base não contém evidência.

03

Separe busca de geração

Registre o top-k recuperado para descobrir se o erro veio do retriever ou do LLM.

04

Compare contra busca simples

Se busca por palavra-chave resolve melhor, você ainda não precisa de uma arquitetura mais complexa.

CHECKPOINT_02

Você entende o RAG como processo, não como pacote.

O próximo passo é implementar um baseline pequeno e medir recuperação, resposta e custo separadamente.
